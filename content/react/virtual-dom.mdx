#### 내용

[출처](https://kentcdodds.com/blog/fix-the-slow-render-before-you-fix-the-re-render)

VirtualDOM이 하는 역할을 하나로 정의하면 'DOM을 업데이트하는 것은 느리니, DOM 업데이트를 묶어서 한번에 처리하자'입니다.

React는 현재의 ReactElement와 렌더링해야 할 ReactElement를 비교합니다. 이를 통해 DOM 업데이트를 수행할 것인지 알 수 있으며, 업데이트를 수행해야 한다면 한번에 묶어서 DOM을 업데이트하고, 이를 "커밋"이라고 부릅니다.

render -> reconciliation -> commit -> state change -> render -> ...

일반적으로 말하는 느린 부분은 리액트 렌더 부분이 아닌 실제 돔이 업데이트되는 "커밋" 단계이며, 모든 DOM 업데이트가 느리다고 할 수도 없습니다.

또한, reconciliation 단계를 통해 실제 돔을 업데이트할지 결정하고 있기에, render가 여러 번 호출된다고 DOM 업데이트가 필연적으로 발생한다고 할 수 없습니다. 흔히 말하는 **불필요한 리렌더링은 DOM이 업데이트되지 않음에도 리액트의 렌더 함수가 호출되는 것**을 이르는데, 일반적으로 이 렌더 단계는 **매우 구형 모바일 기기의 브라우저에서도 빠르게 동작**합니다.

렌더 및 조정 단계가 그렇게 느리지 않다면 앱이 종종 멈추는 이유는 렌더링 단계에서 코드를 실행시키는 과정 중 필연적으로 느리게 동작하게끔 하는 코드가 있을 수 있음을 의미한다.

**프로파일러를 통해 진단 후 해결하는 작업이 필요**합니다. 이런 문제를 리렌더링 횟수를 감소시켜서 해결할 수도 있지만, **원인 자체를 해결한 것은 아니기 때문에 상황이 악화될 가능성이 큽니다**.
